# DAHAB — Application & Database Architecture

> Audience: a database developer who needs to understand how the app is wired
> end-to-end so they can shape, migrate, or extend the underlying schema with
> confidence.
>
> Scope: data model, RLS, RPCs, flows, route → table map, and recommendations.
> No UI styling is described here.

---

## 1. System Overview

DAHAB is a multi-currency holder/ledger application used by staff (admin,
teller, auditor) to manage customer "holders", their per-currency accounts,
vault operations, deposits/withdrawals with approvals, an Excel import
pipeline, account groups for monitoring, and notifications.

**Stack**

- Frontend: TanStack Start v1 (React 19, Vite 7), Tailwind v4, shadcn/ui.
- Backend: Supabase (Postgres + Auth + Storage + Edge Functions), accessed
  from the client via `@/integrations/supabase/client` and from server
  functions via `@/integrations/supabase/client.server`.
- Hosting: Cloudflare Workers (via TanStack Start's Worker target).
- Realtime: Supabase Realtime channels (notifications, ledger updates).
- Cron: `pg_cron` + `pg_net` calling `/api/public/hooks/notifications-tick`.

**Two coexisting data models (important)**

The schema reflects an in-progress consolidation. A DB developer should be
aware of both before changing anything:

| Model | Tables | Purpose | Money type |
|---|---|---|---|
| **Holder model** (primary in current UI) | `account_holders`, `holder_accounts`, `holder_ledger_entries`, `account_groups`, `account_group_members`, `account_name_aliases`, `account_import_*`, `account_link_review_queue` | Customer profiles, per-currency accounts, ledger, groups, import pipeline. | `numeric` |
| **Legacy / vault model** | `accounts`, `account_balances`, `transactions`, `ledger_entries`, `transaction_attachments` | Vault accounts, double-entry transactions, approvals, attachments, audit chain for postings. | `bigint` minor units |

The two are bridged by `ensure_customer_account_for_holder_account(holder_account_id)`
which lazily creates a matching row in `accounts` (kind = `customer`,
nature = `credit`) for any `holder_accounts` row, so vault transactions can
still post against the same holder.

## 2. Authentication & Authorization

- Supabase Auth: email/password and Google OAuth. Anonymous sign-ins are
  **forbidden**. Email confirm is **on** by default.
- Roles enum: `app_role = (admin, teller, auditor, consumer)`.
- Roles live **only** in `public.user_roles` — never on profiles or users.
- Security-definer helpers used by every RLS policy:
  - `has_role(_user_id uuid, _role app_role) returns boolean`
  - `is_staff(_user_id uuid) returns boolean` — true for admin/teller/auditor.
- `handle_new_user()` AFTER INSERT trigger on `auth.users` creates the
  matching `public.profiles` row.
- WebAuthn passkeys: `webauthn_credentials` (per-device) and short-lived
  `webauthn_challenges`. Server functions live in `src/server/webauthn.*`.
- Idle/session timeout enforced client-side (`src/lib/session-timeout.tsx`).

## 3. Enumerated Types & Conventions

**Enums (Postgres)**

- `app_role` — `admin | teller | auditor | consumer`
- `account_kind` — `customer | vault | discount | …`
- `account_nature` — `Debit | Credit` (also lowercase `debit | credit` in legacy `accounts`)
- `vault_channel` — physical channel for vault movements
- `currency_code` / app currency — `LYD | USD | EUR | GBP`
- `tx_status` — includes `pending | posted | rejected | reversed`
- `tx_direction` — `deposit | withdraw`
- `tx_channel` — vault channels for transactions
- `notification_event` — typed event names (`tx_posted`, `pending_created`,
  `approval_decision`, `large_tx`, `low_vault`, `overdraft`,
  `daily_summary`, `account_change`, `reminder_pending`, `reminder_shift`)
- `notification_severity` — `info | warning | critical`

**Identifiers**

- `dahab_account_number` — the holder-level public number, generated by
  `next_dahab_account_number()` from sequence `dahab_holder_seq`.
- `account_number` — per-currency account number on `holder_accounts`,
  generated by `next_holder_account_number()` and uniquely indexed
  (`ux_holder_accounts_account_number`).
- `tx_number` — human-readable transaction reference on `transactions`
  and `holder_ledger_entries`.

**Money representation (NOT yet unified)**

- Legacy: `bigint` minor units (`amount_minor`, `balance_minor`,
  `debit_limit_minor`).
- Holder model: `numeric` (`current_balance`, `debit_amount`,
  `credit_amount`, `balance_after`).
- Recommendation: pick one. See §10.

## 4. Database Schema Reference

Full RLS policies are in the live schema dump that accompanies this doc;
below are the per-table responsibilities, the relationships you must respect,
and the access summary. Standard `id / created_at / updated_at` columns are
omitted unless they encode behavior.

### 4.1 Identity & access

| Table | Purpose | Key columns | Notes |
|---|---|---|---|
| `profiles` | Display name keyed by `auth.users.id`. | `id (uuid, PK = auth.users.id)`, `full_name` | Created by `handle_new_user` trigger. |
| `user_roles` | Role grants. | `(user_id, role)` unique | Only place roles are stored. |
| `webauthn_credentials` | Registered passkeys. | `credential_id`, `public_key`, `counter`, `transports[]` | Insert path is via server function only. |
| `webauthn_challenges` | Short-lived challenges (5 min). | `purpose`, `challenge` | RLS empty — accessed via server function with service role. |

### 4.2 Holders & accounts (primary model)

| Table | Purpose | Key columns |
|---|---|---|
| `account_holders` | Customer profile. | `dahab_account_number` UNIQUE, `canonical_name`, `normalized_name`, `phone`, `email`, `holder_type`, `status`, `owner_user_id` |
| `holder_accounts` | Per-currency account for a holder. | `account_holder_id` → `account_holders.id`, `currency_code`, `account_number` UNIQUE, `account_display_name`, `account_nature`, `current_balance numeric`, `is_primary_account`, `status` |
| `account_name_aliases` | Alternative names for matching/import. | `account_id`, `alias_name`, `alias_type` |
| `holder_ledger_entries` | Per-account ledger lines. | `account_id` → `holder_accounts.id`, `tx_number`, `posted_at`, `currency_code`, `debit_amount`, `credit_amount`, `balance_after` |

### 4.3 Groups (viewing-only monitoring)

| Table | Purpose |
|---|---|
| `account_groups` | Named groups owned by admins. |
| `account_group_members` | Many-to-many of `holder_accounts` ↔ `account_groups` (FK with `ON DELETE CASCADE`). |

Aggregation is computed by `get_group_totals(p_group_id bigint)` which
returns rows of `(currency, total_debits, total_credits, total_balance,
account_count)`.

### 4.4 Excel import pipeline

| Table | Purpose |
|---|---|
| `account_import_batches` | One row per uploaded file. Tracks `total_rows / successful_rows / review_rows / failed_rows / status`. |
| `account_import_staging` | Per-row parsed payload + matching candidates + error message. |
| `account_link_review_queue` | Ambiguous rows promoted from staging that require an admin to confirm/override the suggested holder. |

Import is parsed in `src/lib/account-import.ts`, persisted by
`import_linked_accounts_batch(...)`, surfaced for review at
`/app/import/review`, and finalized by `approve_import_batch(batch_id)` and
`resolve_review_row(row_id, decision)`.

### 4.5 Legacy vault & double-entry transactions

| Table | Purpose |
|---|---|
| `accounts` | Single ledger account: `kind` (customer/vault/discount/…), `nature`, `vault_channel`, `owner_user_id`. Bridged to a `holder_accounts` row via `ensure_customer_account_for_holder_account`. |
| `account_balances` | One row per `(account_id, currency)`: `balance_minor`, `debit_limit_minor`. Updated by `post_transaction`. |
| `transactions` | Header row: `tx_number`, `direction`, `channel`, `currency`, `amount_minor`, `customer_account_id`, `vault_account_id`, `status`, lifecycle timestamps, `created_by_user_id`, `approved_by_user_id`, `comment`, plus correction links `reverses_tx_id` and `corrected_by_tx_id`. |
| `ledger_entries` | Two rows per posted transaction (debit + credit). Append-only. |
| `transaction_attachments` | Files in storage bucket `tx-attachments` (private). |

Posting invariants are enforced by `post_transaction` and the correction
flow uses `correct_transaction` to create a reversal + replacement,
preserving the audit trail.

### 4.6 Reference data

| Table | Purpose |
|---|---|
| `currencies` | `currency_code` PK, `currency_name`, `symbol`. World-readable. |

### 4.7 Notifications

| Table | Purpose |
|---|---|
| `notifications` | Per-user inbox: `event_type`, `severity`, `title`, `body`, `data jsonb`, optional `transaction_id`, `read_at`. |
| `notification_preferences` | Per-user toggles, thresholds (`large_tx_threshold`, `low_vault_threshold`), quiet hours, daily summary time, push opt-in. |
| `notification_reminders_state` | Per-user, per-`kind` last-sent timestamps used by the reminder cron. |
| `push_subscriptions` | Web Push endpoints registered per device. |

Realtime: clients subscribe to topic `notif:<user-id>` on `realtime.messages`
(an RLS policy enforces that the topic prefix matches the caller's uid).

### 4.8 Audit

| Table | Purpose |
|---|---|
| `audit_log` | Append-only log written by sensitive RPCs. Readable by `admin` or `auditor` only. No INSERT/UPDATE/DELETE policy — writes happen via security-definer functions. |

## 5. Key SQL Functions / RPCs

All listed functions live in `public` and are SECURITY DEFINER unless noted.
They are the only sanctioned write paths for the data they touch — the UI
calls them via `supabase.rpc(...)`.

**Auth / role helpers**
- `has_role(_user_id uuid, _role app_role) → bool`
- `is_staff(_user_id uuid) → bool`
- `lookup_user_email_for_credential(text) → text` (passkey login)

**Numbering**
- `next_dahab_account_number() → text`
- `next_holder_account_number() → text`
- `set_account_number()` — BEFORE INSERT trigger on `holder_accounts` that
  fills `account_number` if NULL.

**Holders & accounts**
- `create_holder_with_accounts(...)` — **two overloads exist**; the active
  one inserts `account_holders` plus N `holder_accounts` in one shot. The
  duplicate signature should be dropped (see §10).
- `add_account_to_holder(p_holder_id, p_currency, p_display_name, p_nature, p_alias)`
- `ensure_customer_account_for_holder_account(p_holder_account_id) → uuid`
  — bridge to legacy `accounts`. Always creates `kind='customer'`,
  `nature='credit'` so overdraft logic works.

**Import**
- `import_linked_accounts_batch(p_batch jsonb, p_file_name text)`
- `resolve_review_row(p_row_id, p_decision, p_target_holder_id)`
- `approve_import_batch(p_batch_id)`

**Transactions**
- `post_transaction(p_tx jsonb)` — validates, writes paired `ledger_entries`,
  updates `account_balances`, and (for holder side) writes
  `holder_ledger_entries`.
- `approve_transaction(p_tx_id, p_comment)`
- `reject_transaction(p_tx_id, p_reason)`
- `correct_transaction(p_original_tx_id, p_new_tx jsonb, p_reason)` — admin
  only; reverses + reposts.

**Aggregations (UI reads these)**
- `get_group_totals(p_group_id bigint)`
- `get_holder_currency_totals(p_holder_id bigint)`

**Notifications**
- `_notify_user(...)`, `_notify_role(...)` — internal, not exposed.
- `notifications_mark_read(p_id uuid)`, `notifications_mark_all_read()`
- `run_notification_reminders()` — service-role only, called by cron.

**Triggers**
- `tg_accounts_notify` on `accounts` — emits change notifications.
- `tg_transactions_notify` on `transactions` — emits status-change events.
- `handle_new_user` on `auth.users` — provisions `profiles`.

**Demo seeding (dev only, revoked from public)**
- `seed_demo_ledger`, `_seed_pending_tx`, `_seed_post_tx`, `_upsert_customer`,
  `_upsert_vault`.

## 6. Core Data Flows

### 6.1 Create a holder

```
UI (NewHolderDialog)
  → supabase.rpc("create_holder_with_accounts", {...})
      → INSERT account_holders (auto dahab_account_number)
      → INSERT N holder_accounts (auto account_number per currency)
      → trigger: set_account_number / numbering
  → UI calls ensure_customer_account_for_holder_account on demand
    (lazy bridge to legacy accounts when first transaction is needed)
```

Uniqueness pitfall: `account_holders.dahab_account_number` is UNIQUE; the
sequence must stay aligned with MAX(existing). Migration
`20260506011235_*` realigns `dahab_holder_seq`.

### 6.2 Add a currency account to an existing holder

`AddLinkedAccountDialog` → `add_account_to_holder` RPC. Auto-numbered;
account number is read-only in the UI.

### 6.3 Excel import

```
Upload .xlsx → parsed in src/lib/account-import.ts
  → import_linked_accounts_batch(jsonb)
      → INSERT account_import_batches
      → INSERT account_import_staging (one row per Excel row)
      → ambiguous rows copied into account_link_review_queue
UI /app/import/review
  → resolve_review_row(...)  (admin picks the right holder)
  → approve_import_batch(batch_id)
      → upsert account_holders / holder_accounts from staging
      → mark batch SUCCESS, update counts
```

### 6.4 Deposit / Withdraw

```
Teller (UI /app/transactions/new/{deposit,withdraw})
  → INSERT transactions (status = pending)
  → tg_transactions_notify → _notify_role(admin, 'pending_created')
Admin (UI /app/approvals)
  → approve_transaction(tx_id) | reject_transaction(tx_id, reason)
      → post_transaction:
          - INSERT 2 ledger_entries (debit + credit)
          - UPDATE account_balances for both sides
          - INSERT holder_ledger_entries for the customer side
          - SET transactions.status='posted', posted_at=now()
      → tg_transactions_notify → _notify_user(...)
```

Corrections never UPDATE posted rows: `correct_transaction` inserts a
reversing tx (`reverses_tx_id`) and links the original via `corrected_by_tx_id`.

### 6.5 Group monitoring

Admin creates `account_groups`, then attaches `holder_accounts` as
`account_group_members`. The list and detail pages call
`get_group_totals(p_group_id)` for the per-currency aggregate strip.

### 6.6 Notifications

- DB triggers + RPC bodies call `_notify_user` / `_notify_role` which insert
  rows into `notifications` and (if applicable) into Realtime broadcast.
- `pg_cron` calls `https://<project>/api/public/hooks/notifications-tick`
  on a schedule, which invokes `run_notification_reminders()` for things
  like pending-tx reminders, shift-end reminders, daily summaries.
- Web Push goes out using `push_subscriptions` records.

### 6.7 Audit

`audit_log` is append-only and only readable by admin/auditor. Writes are
performed inside SECURITY DEFINER functions (e.g. corrections, role
changes, import approvals). Do **not** add INSERT policies — keep it
function-mediated.

## 9. Storage & Edge Surfaces

- **Storage bucket** `tx-attachments` (private). Files referenced by
  `transaction_attachments.storage_path`. RLS on `storage.objects` mirrors
  the table policy: staff or uploader or owner of the linked transaction.
- **Realtime**: explicit policy on `realtime.messages` requires the topic
  to start with `notif:<auth.uid()>` (or contain the uid) to subscribe.
- **Public API routes** under `src/routes/api/public/*` bypass auth
  middleware on the published site; each handler MUST validate a shared
  secret/signature before doing any work.
- **Server functions** live in `src/server/<feature>.functions.ts` (typed
  RPC wrappers via `createServerFn`) and `src/server/<feature>.server.ts`
  (server-only helpers using the service-role client).

## 10. Recommendations for the DB Developer

1. **Unify money representation.** Pick one of:
   - `numeric(20,4)` everywhere (simpler reporting, matches holder model), or
   - `bigint` minor units everywhere (better for exact arithmetic).
   Then write a one-shot migration to convert the other model and update
   the affected RPCs (`post_transaction`, `get_group_totals`, …).
2. **Drop the duplicate `create_holder_with_accounts` overload.** Two
   signatures are currently registered; PostgREST will reject ambiguous
   calls. Keep the overload that takes the JSON array of accounts.
3. **Add missing FOREIGN KEYs.** The schema dump shows several tables
   without declared FKs (`holder_accounts.account_holder_id`,
   `holder_ledger_entries.account_id`, `account_import_staging.import_batch_id`,
   `account_link_review_queue.staging_id`, `account_name_aliases.account_id`,
   etc.). Add them with explicit `ON DELETE` policies (CASCADE for
   children of import batches; RESTRICT for ledger lines).
4. **One account per (holder, currency)?** If that is the business rule,
   add `UNIQUE (account_holder_id, currency_code)` on `holder_accounts`.
5. **Indexes for hot queries**:
   - `holder_ledger_entries (account_id, posted_at DESC)`
   - `transactions (status, posted_at DESC)`
   - `transactions (customer_account_id, posted_at DESC)`
   - `notifications (user_id, created_at DESC) WHERE read_at IS NULL`
   - `account_import_staging (import_batch_id, review_status)`
6. **Sequence drift.** `dahab_holder_seq` has had to be realigned once
   already (migration `20260506011235`). Consider a CHECK / trigger that
   re-seeds the sequence on bulk import, or switch the public number to a
   formatted `bigint` derived from `id`.
7. **Audit chain.** Keep `audit_log` insert-blocked at the policy layer;
   route every write through SECURITY DEFINER functions so the actor and
   target are recorded consistently.
8. **Long term.** Plan a deprecation of the legacy `accounts /
   account_balances / ledger_entries / transactions` set in favour of a
   single ledger built on `holder_accounts` + `holder_ledger_entries`,
   keeping vault accounts as a distinct `holder_accounts.kind` (or a
   sibling table) and posting through a unified `post_transaction_v2`.

### C. Glossary

- **DAHAB number** — public-facing identifier of a holder
  (`account_holders.dahab_account_number`).
- **Holder** — a customer entity that owns one or more currency accounts.
- **Holder account** — one (holder, currency) account, with its own
  `account_number` and ledger.
- **Vault** — internal `accounts.kind = 'vault'` row that represents a
  cash/liquidity store; the counterparty of every customer movement.
- **Channel** — physical or logical route a vault movement uses
  (`vault_channel`).
- **Nature** — `Debit` or `Credit`. Drives sign convention and overdraft
  logic.
- **Ledger entry** — one side of a double-entry posting; two per
  transaction in the legacy model, one per posting per account on the
  holder side.
- **Bridge** — `ensure_customer_account_for_holder_account` linking a
  holder account to a legacy `accounts` row so vault transactions can
  post against it.
- **Group** — viewing-only collection of holder accounts an admin wants
  to monitor together. No financial linkage between members.

## Backend swap plan (AWS Lambda)

The frontend talks to backend auth via the `AuthService` interface in
`src/lib/authService.ts`. Switch implementations with the build-time env
var `VITE_AUTH_BACKEND` (`supabase` default, `lambda` when ready).

### REST contract the Lambda backend must implement

| Method | Path | Body | Response |
| --- | --- | --- | --- |
| POST | `/auth/sign-in` | `{ email, password }` | `{ userId, email, mustChangePassword, token }` |
| POST | `/auth/forgot-password` | `{ email }` | `204` (always — no enumeration) |
| POST | `/auth/reset-password` | `{ token, newPassword }` | `204` |
| POST | `/auth/me/clear-must-change-password` | _none_ | `204` |
| POST | `/auth/sign-out` | _none_ | `204` |
| GET  | `/users/{userId}/must-change-password` | _none_ | `{ mustChangePassword: boolean }` |
| POST | `/admin/users/{userId}/reset-password` | _none_ | `{ ok, email }` (admin only — revokes sessions + emails reset link) |

`mustChangePassword` is part of the user profile/JWT claim. When `true`,
the frontend redirects to `/change-password` immediately after sign-in.

Base URL is read from `VITE_LAMBDA_API_BASE_URL`. The stub in
`src/lib/authService.lambda.ts` documents the exact call shape per method.
